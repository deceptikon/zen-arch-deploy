# arch-deploy тАФ Reproducible Arch Linux Deployment

Machine-specific, auditable, dry-run-safe Arch Linux deployment pipeline.
Built for an ASUS Zenbook 14 (UEFI + BTRFS), but generic enough for any compatible machine.

## Quick Start

### Phase 1 тАФ On your current running system

```bash
# 1. Inspect the current system state
sudo ./arch-deploy.sh inspect

# 2. Generate a machine profile from the inspection data
sudo ./arch-deploy.sh generate-profile

# 3. Review and edit the generated profile
#    Key field to check: profiles/my-machine.yaml
#    - Verify storage.disk, storage.partitions.efi.device, storage.partitions.root_pool.device
#    - Set dotfiles.repo if you want to pull from a remote repo (optional тАФ bundled dotfiles are used by default)
vim profiles/my-machine.yaml

# 4. Commit your profile to the repo before wiping
git add profiles/my-machine.yaml && git commit -m "add my-machine profile"
git push
```

### Phase 2 тАФ From the Arch ISO (live USB)

```bash
# Boot the Arch ISO, connect to network, then:
git clone https://github.com/deceptikon/zen-arch-deploy.git
cd zen-arch-deploy

# Profile is auto-detected if exactly one non-example profile exists in profiles/
# Or pass it explicitly: ./arch-deploy.sh --profile profiles/my-machine.yaml <stage>

sudo ./arch-deploy.sh validate         # Read-only security audit
sudo ./arch-deploy.sh prepare --dry-run
sudo ./arch-deploy.sh prepare
sudo ./arch-deploy.sh execute --dry-run
sudo ./arch-deploy.sh execute          # тЪая╕П Destructive тАФ resets @ subvolume
```

> **After `execute` completes**, a copy of the entire `arch-deploy` directory (including your generated profile) is automatically placed at `/root/arch-deploy` on the new system. No need to clone again after rebooting.

### Phase 3 тАФ After first boot into the new system

```bash
# Log in as root, then:
cd /root/arch-deploy

# If you didn't have a profile (it was lost), regenerate it first:
# sudo ./arch-deploy.sh inspect && sudo ./arch-deploy.sh generate-profile

sudo ./arch-deploy.sh configure       # Creates user, installs AUR packages, deploys dotfiles
sudo ./arch-deploy.sh verify          # Health checks and baseline Timeshift snapshot
```

## Stages

| # | Stage | Environment | Destructive? |
|---|-------|-------------|--------------|
| 01 | **inspect** | Current host | No |
| 02 | **generate-profile** | Current host | No |
| 03 | **validate** | Arch ISO | No (read-only) |
| 04 | **prepare** | Arch ISO | No |
| 05 | **execute** | Arch ISO | **Yes** (resets `@` subvol) |
| 06 | **configure** | Installed system (root) | No |
| 07 | **verify** | Installed system | No |

## Wipe Strategy

- **`subvol-reset`** (default on existing BTRFS): Renames `@` to `@.bak-<timestamp>`, then creates a fresh `@`. `@home`, `@swap`, and the EFI partition are fully preserved.
- **`format-full`** (generated for non-BTRFS or fresh disks): Formats the EFI partition as FAT32 and root as BTRFS, then creates all subvolumes from scratch.

## Profile Auto-Detection

The orchestrator (`arch-deploy.sh`) will automatically detect your profile if exactly one non-example profile exists in `profiles/`. You do not need to pass `--profile` in the common case. All stage scripts also fall back to the `$ARCH_DEPLOY_PROFILE` environment variable exported by the orchestrator.

## Dotfiles

Dotfiles are managed by [chezmoi](https://www.chezmoi.io/) and live in the `dotfiles/` directory inside this repository (monorepo).

- **Stage 06 (configure)** automatically deploys dotfiles to the new user's home directory.
- If `dotfiles.repo` is **not set** in your profile (the default), the bundled `dotfiles/` directory is copied directly to `~/.local/share/chezmoi` and applied.
- If `dotfiles.repo` **is set** to an external URL, `chezmoi init --apply --force <repo>` is used instead.

## Pipeline Design Rules

1. **No `/dev/null` silencing.** Every command's output is either captured or logged. Errors must be visible.
2. **No `|| true` masking.** Failures are handled with explicit `if/else` branches.
3. **Boundary validation between stages.** `inspect_validate()` must pass before `generate-profile` runs. `profile_validate()` must pass after every `profile_load()`.
4. **`set +e` in data collectors.** `01-inspect.sh` and `02-generate-profile.sh` run without `errexit` тАФ one failed `pacman -Q` should not abort the entire scan.
5. **Always `--dry-run` first.** `execute` requires typing `yes` before any destructive operation.

## VM Test (QEMU/KVM)

Test the entire pipeline in a virtual machine before touching the real system.

### Prerequisites

```bash
sudo pacman -S qemu-full edk2-ovmf parted dosfstools btrfs-progs
```

### Run

```bash
# Create virtual disk and launch VM with Arch ISO
./vm-test/run-qemu.sh

# Clean disk and restart
./vm-test/run-qemu.sh --clean && ./vm-test/run-qemu.sh
```

### Inside the VM

```bash
git clone https://github.com/deceptikon/zen-arch-deploy.git && cd zen-arch-deploy

# Use the VM-specific profile (targets /dev/vda, format-full strategy)
sudo ./arch-deploy.sh --profile profiles/zenbook-vm.yaml prepare --dry-run
sudo ./arch-deploy.sh --profile profiles/zenbook-vm.yaml prepare
sudo ./arch-deploy.sh --profile profiles/zenbook-vm.yaml execute --dry-run
sudo ./arch-deploy.sh --profile profiles/zenbook-vm.yaml execute

# Reboot, then:
sudo ./arch-deploy.sh --profile profiles/zenbook-vm.yaml configure
sudo ./arch-deploy.sh --profile profiles/zenbook-vm.yaml verify
```

### Success Criteria

| Check | Expected |
|-------|----------|
| `findmnt /` | Shows `subvol=@` |
| `findmnt /home` | Shows `subvol=@home` (preserved) |
| `systemctl is-enabled sddm` | `enabled` |
| `efibootmgr` | Shows `ARCHLINUX` entry |
| `timeshift --list` | Shows at least one snapshot |

### Troubleshooting

| Problem | Fix |
|---------|-----|
| No network in VM | Run `dhcpcd` inside the VM |
| Cannot see EFI partition | Ensure UEFI boot, not legacy BIOS |
| Wrong disk device | VM disk is `/dev/vda`, real machine is `/dev/nvme0n1` тАФ use correct profile |
| `pacman: file conflicts` | Already handled тАФ pipeline uses `--overwrite '*'` |
